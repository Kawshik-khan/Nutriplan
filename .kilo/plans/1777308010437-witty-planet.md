# Nutriplan Data Flow Repair Plan

## Current Architecture Assessment

**Frontend:**
- React + Vite, TanStack Query for server state
- Client-side `meal-context.tsx` using localStorage for meal planning
- No backend integration for meal planning

**Backend:**
- Express + Prisma (SQLite)
- Existing models: User, Meal (logged meals), NutritionStat, WaterLog, WeightLog
- Missing models: planner_entries, food_logs, meals (database meals)

**Issues:**
- Meal planner operates in isolation with localStorage
- Dashboard shows hardcoded stats
- No synchronization between components
- Missing database tables and APIs for meal planning
- Food log uses hardcoded data

## Database Schema Updates

### New Tables Required
1. **meals** - Database meals/foods catalog
2. **planner_entries** - Meal planner entries
3. **food_logs** - Daily food consumption logs

### Schema Changes
```prisma
model Meal {
  id        Int      @id @default(autoincrement())
  name      String
  image     String?
  calories  Int
  proteinG  Int
  carbsG    Int
  fatG      Int
  category  String?  // Breakfast, Lunch, Dinner, Snack
  tags      String?
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  
  // Relations
  plannerEntries PlannerEntry[]
  foodLogs       FoodLog[]
}

model PlannerEntry {
  id        Int      @id @default(autoincrement())
  userId    Int
  mealId    Int
  date      DateTime // Full date for planning
  mealType  String   // Breakfast, Lunch, Dinner, Snack
  quantity  Int      @default(1)
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  
  user User @relation(fields: [userId], references: [id], onDelete: Cascade)
  meal Meal @relation(fields: [mealId], references: [id], onDelete: Cascade)
  
  @@unique([userId, date, mealType, mealId])
  @@index([userId, date])
}

model FoodLog {
  id        Int      @id @default(autoincrement())
  userId    Int
  mealId    Int?     // Reference to meals table
  foodName  String?  // For custom foods not in meals table
  date      DateTime // Full date for logging
  mealType  String   // Breakfast, Lunch, Dinner, Snack
  quantity  Int      @default(1)
  calories  Int
  proteinG  Int
  carbsG    Int
  fatG      Int
  createdAt DateTime @default(now())
  
  user User  @relation(fields: [userId], references: [id], onDelete: Cascade)
  meal Meal? @relation(fields: [mealId], references: [id], onDelete: SetNull)
  
  @@index([userId, date])
}
```

## Backend API Development

### New API Endpoints
1. **Meals Management** (`/api/meals`)
   - GET `/api/meals` - List all meals with search/filter
   - GET `/api/meals/:id` - Get meal details

2. **Meal Planner** (`/api/planner`)
   - GET `/api/planner/:date` - Get meals for specific date
   - POST `/api/planner` - Add meal to planner
   - DELETE `/api/planner/:entryId` - Remove meal from planner

3. **Food Log** (`/api/food-log`)
   - GET `/api/food-log/:date` - Get logged meals for date
   - POST `/api/food-log` - Log food consumption
   - DELETE `/api/food-log/:logId` - Remove logged food

4. **Dashboard** (Update existing)
   - Update `/api/dashboard/overview` to calculate from planner_entries and food_logs

### State Management Overhaul

#### Replace Client-Side Context with Server-Sync
1. **Global State Store** (Zustand/Redux Toolkit)
   - `meals`: Database meals catalog
   - `plannerEntries`: Current week's planner entries
   - `foodLogs`: Today's logged foods
   - `dailyNutrition`: Calculated nutrition stats

2. **Real-Time Sync**
   - TanStack Query for server state
   - Optimistic updates for better UX
   - Background refetch on focus/tab changes
